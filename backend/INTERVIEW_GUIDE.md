# 面试材料

## 1 分钟项目介绍

这是一个面向求职者的 AI 面试系统，包含刷题练习、简历上传、AI 模拟面试、动态追问、Topic 切换和结构化面试报告。用户可以先通过刷题模块练习技术问题，再上传简历进入 AI 面试。系统会根据简历生成面试 Topic 和第一题，根据回答质量动态决定继续追问还是切换 Topic，面试结束后生成综合评分、Topic 分析、优缺点和复习建议。项目使用 Next.js、FastAPI、MySQL、LangChain 和 Ollama，重点展示了 AI 应用开发中的 Prompt 编排、结构化输出解析、状态管理、数据持久化和自动化测试。

## 3 分钟项目介绍

这个项目的核心目标是解决求职者准备技术面试时缺少针对性反馈的问题。传统刷题系统通常只给固定答案，普通聊天机器人又缺少流程控制和复盘能力，所以我设计了一个完整的 AI 面试助手。

系统分为三条主链路。

第一条是刷题链路。用户输入题库、题型和技术方向，后端调用 LLM 生成题目。用户提交答案后，系统从技术深度、逻辑表达、项目经验和沟通表达几个维度评分，并保存刷题记录。

第二条是 AI 面试链路。用户上传简历后，后端解析 PDF、DOCX 或 TXT，基于简历生成面试 Topic 和第一题。用户每次回答后，系统保存上下文，再由 LLM 判断是继续追问还是切换 Topic。为了避免体验僵硬，我没有固定追问次数，而是设计了动态 Topic 切换机制，并在代码层做兜底，比如高分提前切换、低分切换、最多追问三次、明显不会直接切换。

第三条是报告链路。面试结束后，系统汇总所有 Topic、消息和评分，生成结构化面试报告，包括综合评分、各 Topic 得分、优势、不足、改进建议和三天复习计划。

技术上，前端使用 Next.js 和 TypeScript，后端使用 FastAPI、SQLAlchemy 和 MySQL，AI 侧使用 LangChain 和 Ollama。项目还补充了 pytest 和 Playwright E2E，覆盖刷题、面试记录、报告展示、异常场景等关键链路。

## 项目亮点

- 简历驱动面试，不是固定题库问答。
- Topic 动态切换，避免固定追问 5 次的僵硬体验。
- cannot_answer 检测，用户明显不会时不继续强行追问。
- 无效数字回答低分处理，避免 LLM 误判高分。
- 结构化报告生成，支持面试复盘。
- 刷题和面试双闭环。
- 前后端完整实现，有鉴权、记录、删除和测试。

## 技术难点

### 1. LLM 输出不稳定

问题：LLM 有时不返回合法 JSON，或分数不合理。

解决：

- Prompt 中明确要求 JSON。
- 代码层解析 JSON。
- 解析失败时兜底。
- 对无效回答、纯数字回答、cannot_answer 做规则修正。

### 2. Topic 切换不能僵硬

问题：固定追问次数会导致体验差。

解决：

- LLM 返回 action、score、reason、next_question。
- 代码层根据 score、follow_up_count 和 cannot_answer 做最终决策。

### 3. 刷题记录和面试记录不能串表

问题：刷题 `interviews.id` 和 AI 面试 `ai_interview_info.id` 容易混用。

解决：

- 刷题使用 `interviews` 和 `interview_answers`。
- AI 面试使用 `ai_interview_info`、`interview_topics`、`interview_messages`、`interview_reports`。
- 接口层校验记录归属和类型。

### 4. 用户数据隔离

问题：不同用户不能看到彼此的面试记录或报告。

解决：

- 所有关键接口依赖 JWT。
- 查询时按 `user_id` 过滤。
- 报告、follow_up、记录删除都做用户归属校验。

## 项目难点

- Prompt 设计要兼顾准确性和结构化输出。
- 面试流程有状态，需要保存 Topic、消息、追问次数。
- LLM 评分可能波动，需要规则兜底。
- 前端聊天 UI 需要处理持续对话和面试结束状态。
- 测试需要 mock LLM，避免真实模型导致自动化测试不稳定。

## 为什么使用 FastAPI

- FastAPI 对 Pydantic 支持好，适合定义请求和响应结构。
- 异步能力和性能适合 AI 接口服务。
- 自动生成 Swagger 文档，方便调试。
- 代码结构清晰，适合拆分 routes、services、schemas。
- 和 Python AI 生态结合自然，比如 LangChain、Ollama、Chroma、pdfplumber。

## 为什么设计 Topic 切换

真实技术面试不会对一个候选人固定追问同一个方向 5 次。面试官会根据回答质量动态判断：

- 回答很好，说明已掌握，可以换方向。
- 回答不会，继续追问没有意义。
- 回答一般，可以再追问细节。

所以系统用 LLM 判断加代码兜底的方式模拟真实面试官决策。

## 如何判断不会回答

系统不会简单判断是否包含“不会”。因为正常回答中也可能出现“不会一直查库”“不会导致缓存击穿”。

判断逻辑：

- 回答很短。
- 内容主要表达不知道、不会、不清楚、没接触过。
- 不包含 Redis、缓存、数据库、接口、方案、优化等技术关键词。

一旦判定 cannot_answer，系统会给低分并切换 Topic。

## 如何生成 AI 报告

报告生成流程：

1. 查询面试记录。
2. 校验用户权限。
3. 校验面试是否完成。
4. 查询 Topic 和全部消息。
5. 汇总候选人回答。
6. 调用 LLM 生成结构化 JSON。
7. 代码层修正分数和无效回答。
8. 保存到 `interview_reports`。
9. 返回给前端展示。

## 面试官可能追问的问题与标准回答

### Q1：这个项目和普通 ChatGPT 对话有什么区别？

普通 ChatGPT 是自由对话，没有业务状态。这个项目有完整面试状态，包括简历、Topic、消息、追问次数、结束状态和报告。它不是单次问答，而是一个有流程控制和数据持久化的 AI 应用。

### Q2：为什么不完全相信 LLM 的评分？

LLM 评分会波动，也可能被示例诱导。因此我加了代码规则兜底，比如纯数字回答直接低分、cannot_answer 直接切换、最高追问次数限制、报告分数按 Topic 和完整度综合计算。

### Q3：Topic 切换的规则是什么？

LLM 输出 action 和 score。代码层判断：

- action 为 switch_topic 切换。
- score 大于等于 85 提前切换。
- score 小于等于 55 且已经追问过则切换。
- follow_up_count 达到 3 强制切换。
- cannot_answer 强制切换。

### Q4：如何保证用户数据安全？

接口使用 JWT 鉴权。查询、删除、生成报告、follow_up 都会校验 `user_id`，确保只能访问自己的数据。

### Q5：如果 LLM 返回非法 JSON 怎么办？

服务层会捕获解析异常，并使用兜底逻辑。例如 follow_up 解析失败时继续追问但不超过最大次数，报告解析失败时生成保守报告，不让系统崩溃。

### Q6：为什么刷题和 AI 面试用了不同表？

两者业务状态不同。刷题是题目和答案评分，适合 `interviews` 和 `interview_answers`。AI 面试有 Topic、消息、报告和状态机，所以单独使用 `ai_interview_info` 等表，避免 ID 串用和业务混乱。

### Q7：项目还有哪些可以继续优化？

- 引入 Alembic 管理数据库迁移。
- 补齐 Docker 和 CI/CD。
- 增加真实题库表。
- 接入更稳定的模型服务。
- 增加更细粒度的报告可视化。
- 做生产级日志、监控和限流。
